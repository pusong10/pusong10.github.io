有限状态机（FSM）在复杂工业控制逻辑中的应用

1. 业务背景与挑战

在处理串口或 TCP 流式数据时，开发者常遇到“粘包”或“断包”。传统实现使用 List<byte> 配合 RemoveRange 不断裁切缓冲区。这种方式存在三大痛点：

  • 内存抖动： 频繁创建数组和挪动数据，触发频繁的 GC。
  • 容错差：遇到干扰产生的杂讯时，往往会丢弃整包数据。
  • 逻辑耦合：解析逻辑混杂在数据接收循环中，难以维护。

有限状态机（FSM） 将解析逻辑拆解为相互独立的原子状态，不仅解决了上述问题，还极大提升了系统的稳定性。

2. 核心设计：6 状态

针对 Modbus RTU/TCP 协议，设计了一下 6 种状态：

  1. IDLE：等待状态起始。
  2. HEADER：处理站号、功能码（及 TCP 的 MABP 头）。
  3. LENGTH：捕获动态载荷长度（PDU LENGTH）。
  4. PAYLOAD：高频采集核心数据区。
  5. CRC：执行 RTU 校验（TCP 模式则跳过）。
  6. COMPLETED：逻辑闭环，触发解析事件。

4. 总结

  • 零 GC 压力：
    通过预分配 _buffer 和 _floatBuffer，在整个解析生命周期内没有新对象的分配。对于超高频轮询任务至关重要。

  • 零拷贝思想：
    在 ParseFrame 阶段，我们使用 Span<byte> 引用原始缓冲区。这意味着从物理链路接收到数据到解析为 float，数据没有经过多余的内存复制。

  • 智能容错（Shift机制）：
    当 CRC 校验失败或长度非法时，系统不是简单地丢弃整包，而是通过 Shift() 进行单字节滑动检测。这保证了在电磁干扰严重的 RS485 总线上，系统具有极强的抗噪能力。
